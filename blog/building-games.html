What Building 6 Games Taught Me About Writing Better Code

February 2026 · 6 min read

Building games is the best education I've had as a developer. Not because games are glamorous, but because they lie to you less. A bug in a to-do app might go unnoticed for weeks. A bug in a game is immediately, painfully visible. The snake goes through a wall. The alien doesn't shoot. The tile doesn't merge. You feel it instantly.

Over the past year I built six games — Snake, 2048, Space Invaders, a Crossword Puzzle, a Quiz Game, and a Pixel Runner platformer. Each one taught me something different. Here's what I took from each of them.

The games and what they taught me

Snake — State machines are underrated

Snake looks trivial. It isn't. Managing the direction the snake is moving, the direction it's about to move, and preventing it from reversing into itself requires careful state management. I learned to think in terms of "what state am I in now, and what's the valid next state?" — a pattern I now use everywhere.

2048 — Elegant algorithms exist

The tile merge logic in 2048 can be written in about 15 lines if you think about it the right way. My first attempt was 80 lines of nested conditionals. When I finally found the clean version — collapsing each row into an array, filtering, merging, padding — it felt like a revelation. Simplicity is a skill.

Space Invaders — Collision detection is a rabbit hole

I rewrote the collision system three times. Pixel-perfect collision is expensive. Bounding-box collision is fast but imprecise. I ended up with a layered approach — cheap broad-phase checks first, more precise checks only when needed. That's a pattern straight out of game engine design, and I stumbled into it by necessity.

Crossword — Data structure design matters early

A crossword is a grid where letters intersect. How you model that in code shapes everything. I chose wrong at the start — a flat array when I needed a 2D map with metadata on each cell. Refactoring halfway through cost me two full days. Now I spend more time on data design before writing any logic.

Quiz Game — User experience is part of the code

The quiz game has 60 questions across 6 categories. Getting the questions right was 10% of the work. The other 90% was transitions, timing, score animations, feedback on wrong answers, and the final results screen. How something feels is just as much a code problem as how it works.

Pixel Runner — requestAnimationFrame is your friend

My first attempt used setInterval for the game loop. It was choppy, inconsistent, and drained the battery. Switching to requestAnimationFrame — and calculating delta time to keep the game speed consistent regardless of frame rate — made it feel like a completely different game. The browser API exists for a reason.

The thing all of them had in common

Every single game went through the same lifecycle: it worked, then it got complex, then I had to refactor. The code that gets you to "working" is rarely the code you want to ship. The difference between a game that's done and a game that's good is usually one honest rewrite of the messiest part.

Don't be precious about code you've already written. If you can see a cleaner way, take it. The first version was the prototype. Now build the real thing.

Why I'd recommend building games to any developer

Games compress the feedback loop. You write code, you see the result immediately, you feel whether it's right. That tight cycle builds intuition faster than anything else I've tried.

They also force you into problems you'd never deliberately seek out — physics, rendering, input handling, AI pathfinding — and those problems teach you more than another CRUD app ever would. Pick a game you enjoy, build a version of it, and don't stop when it works. Stop when it feels good.
