How I Built a Live Weather App (And What Nearly Broke Me)

March 2026 · 5 min read

When I decided to build a weather app, I thought it would be simple. Fetch some data, display it nicely, done. Tutorials make it look that way. Reality was a bit different.

The finished app has live forecasts, hourly data, a 7-day outlook, sunrise and sunset times, UV index, wind speed, humidity, and city search with auto-suggest. Getting all of that working — and looking good — took a lot more trial and error than I expected. Here's the honest account of how it went.

Starting with the API

I used the OpenWeatherMap API, which has a generous free tier. Setting it up looked easy: register, get a key, make a fetch call, get weather. That part worked first time. What didn't work first time was everything after that.

Real API data is messy. Some fields you expect are sometimes missing. The temperature comes back in Kelvin by default (didn't notice that for longer than I'd like to admit). Wind direction is in degrees, not compass points — so I had to write a function to convert 247° into "WSW". UV index only comes from a separate endpoint. Sunrise and sunset are Unix timestamps that need converting.

Every tutorial shows you the happy path. Real data has null values, unexpected formats, and edge cases the documentation glosses over. Plan for that from the start.

The CORS problem

For a while, my app worked perfectly when I opened it locally — and broke completely when I pushed it live. The browser was blocking requests to the API because of CORS (Cross-Origin Resource Sharing) restrictions.

The fix was switching from the HTTP endpoint to HTTPS and adding the right headers. Took me a full afternoon to figure that out. The error message in the console wasn't exactly clear, and every Stack Overflow answer seemed to suggest something slightly different.

Making it look good

Once the data was working, I spent a long time on the UI. Weather apps live or die on how they feel — nobody wants a list of numbers. I ended up building:

  • A dynamic background that shifts based on weather condition and time of day
  • Custom SVG icons for each weather state (sun, cloud, rain, snow, storm)
  • A horizontal scroll for the hourly forecast
  • Cards for the 7-day outlook that show min/max temps at a glance

The background gradient took the longest. I wanted it to feel atmospheric — a deep orange sunrise, a steel-grey overcast, a dark navy night. Getting the colour transitions right without it looking garish was genuinely difficult.

The city search

The search feature uses a separate geocoding API call to convert a city name into coordinates, then passes those to the weather API. Sounds straightforward. In practice, you have to handle:

  • Cities with the same name in different countries
  • Empty results when the city isn't found
  • Rate limiting if someone types fast
  • Graceful fallback to the user's location if search fails

I added a debounce to the search input — it waits 400ms after the user stops typing before firing the request. That alone cut the number of API calls by about 80% in testing.

What I'd do differently

If I was starting again, I'd plan the data model first. I dived straight into building the UI before I fully understood what data I had available and in what format. That led to a lot of refactoring mid-build when I realised something I'd assumed was a single call actually needed three.

The app is live on the Work tab — give it a try. It'll ask for your location and pull in the live forecast wherever you are.
